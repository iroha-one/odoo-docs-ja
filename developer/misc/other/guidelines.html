Odoo ガイドライン¶

本ページでは、Odoo コーディングガイドラインを紹介します。Odoo Appsコードの品質向上を目的としています。 確かに、適切なコードは、読みやすさを改善し、メンテナンスを容易にし、デバッグを助け、複雑さを減らし、信頼性を高めます。 これらのガイドラインは、すべての新しいモジュールとすべての新しい開発に適用されるようにします。

警告

安定版 で既存のファイルを変更する場合、オリジナルのファイルスタイルは他のスタイルガイドラインより厳密に優先されます。 つまり、これらのガイドラインを適用するために、既存のファイルを変更しないでください。 コード行のリビジョン履歴を乱すのを避けます。差分は最小限に抑えるべきです。詳細については、 プルリクエストのガイド を参照してください。

警告

master (development) バージョン の既存のファイルを変更する場合は、既存のコードにこれらのガイドラインを適用します。 つまり、既存のファイル構造を大幅に変更する場合にのみ適用します。 その場合、最初に move コミットを実行し、その機能に関連する変更を適用します。

モジュール構造¶

ディレクトリ¶

モジュールは重要なディレクトリで構成されています。 それらはビジネスロジックを含んでいますので、それを見れば、モジュールの目的を理解することとなるでしょう。

  • data/ : デモ・データと データ用 xml

  • models/ : モデル定義

  • controllers/ : コントローラーが含まれています (HTTP routes)

  • views/ : ビューとテンプレートが含まれています。

  • static/ : Web アセットを含んでいます。 css/, js/, img/, lib/, ... と分けられています。

他にも、任意のディレクトリでモジュールを構成します。

  • wizard/ : トランジェントモデル (models.TransientModel) とそのビューをを再編成しています。

  • report/ : 印刷可能なレポートとSQL ビューに基づくモデルが含まれています。Python オブジェクトと XML ビューは、このディレクトリに含まれています。

  • tests/ : Python テストが含まれています。

ファイルの命名¶

ファイルの命名は、すべてのアドオンから情報をすばやく見つけるために重要です。このセクションでは、標準の odoo モジュールでファイルに名前を付ける方法を説明します。 例として、plant nursery アプリケーションを使用します。 このアプリケーションでは、plant.narsery と plant.order といった2つの主要なモデルを保持しています。

models については、同じメインモデルに属するモデルでビジネスロジックを分割します。 各セットは、メインモデルに基づいて指定されたファイルにあります。 モデルが 1 つだけの場合は、モジュール名と同じ名前になります。 それぞれの継承されたモデルは、影響を受けたモデルの理解に役立つ独自のファイルでなければなりません。

addons/plant_nursery/
|-- models/
|   |-- plant_nursery.py (first main model)
|   |-- plant_order.py (another main model)
|   |-- res_partner.py (inherited Odoo model)

Concerning security, three main files should be used:

  • First one is the definition of access rights done in a ir.model.access.csv file.

  • User groups are defined in <module>_groups.xml.

  • Record rules are defined in <model>_security.xml.

addons/plant_nursery/
|-- security/
|   |-- ir.model.access.csv
|   |-- plant_nursery_groups.xml
|   |-- plant_nursery_security.xml
|   |-- plant_order_security.xml

views については、バックエンドビューをモデルと同様に分割し、サフィックスとして _views.xml を付ける必要があります。バックエンドビューには、リストビュー、フォームビュー、カンバンビュー、アクティビティビュー、グラフビュー、ピボットビュー...などがあります。ビューのモデルごとの分割を容易にするために、特定のアクションにリンクしていないメインメニューは、任意で <module>_menus.xml ファイルに抽出することができます。テンプレート (主にポータルやウェブサイトの表示に使われる QWeb ページ) とバンドル (JS および CSS アセットのインポート) は、それぞれ別のファイルに格納されます。それぞれ <model>_templates.xml と assets.xml ファイルです。

addons/plant_nursery/
|-- views/
|   | -- assets.xml (import of JS / CSS)
|   | -- plant_nursery_menus.xml (optional definition of main menus)
|   | -- plant_nursery_views.xml (backend views)
|   | -- plant_nursery_templates.xml (portal templates)
|   | -- plant_order_views.xml
|   | -- plant_order_templates.xml
|   | -- res_partner_views.xml

data については、目的(デモまたはデータ)とメインモデルで分割します。ファイル名は _demo.xml または _data.xml でサフィックスされた main_model 名になります。 たとえば、メインモデルとサブタイプのデモとデータを持つアプリケーションの場合。 アクティビティとメールテンプレートはすべてメールモジュールに関連しています:

addons/plant_nursery/
|-- data/
|   |-- plant_nursery_data.xml
|   |-- plant_nursery_demo.xml
|   |-- mail_data.xml

controllers については、一般的にすべてのコントローラは、 <module_name>.py というファイルに含まれる1つのコントローラに属しています。Odoo の古い慣習では、このファイルを main.py と名付けていますが、これは時代遅れと考えられます。既存のコントローラを他のモジュールから継承する必要がある場合は、 <inherited_module_name>.py で行います。例えば、アプリケーションにポータルコントローラを追加するには、 portal.py を使用します。

addons/plant_nursery/
|-- controllers/
|   |-- plant_nursery.py
|   |-- portal.py (inheriting portal/controllers/portal.py)
|   |-- main.py (deprecated, replaced by plant_nursery.py)

static files については、JavascriptファイルでPythonモデルと同じロジックに従います。 各コンポーネントは、意味のある名前のファイルに含める必要があります。たとえば、メールモジュールの activity.js にアクティビティウィジェットがあります。 サブディレクトリは 'package' を構造化するために作成することもできます (詳細は web モジュールを参照してください)。 JSウィジェットのテンプレート(静的XMLファイル)とスタイル(scssファイル)に同じロジックを適用する必要があります。 Odoo の外にあるデータ (画像、ライブラリ) をリンクしないでください: URL を画像に使用せず、代わりにコードベースにコピーしてください。

wizards については、命名規則はPythonモデルと同じです: <transient>.py と <transient>_views.xml です。 両者ともウィザードディレクトリに配置されています。この名前は、トランジェントモデルのウィザードキーワードを使用している古いodooアプリケーションから来ています。

addons/plant_nursery/
|-- wizard/
|   |-- make_plant_order.py
|   |-- make_plant_order_views.xml

XML ファイル¶

フォーマット¶

XMLでレコードを宣言するには、 record ( <record> を使用) 記法を推奨します。

  • model の前に id 属性を配置します

  • フィールド宣言の場合、name 属性が最初です。 次に、 field タグか eval 属性のどちらかに value を配置し、そして最後に重要性によって順序付けられた他の属性 (ウィジェット、オプション、...) を配置します。

  • モデル別にレコードをグループ化してみてください。アクション/メニュー/ビュー間の依存関係がある場合、この規則は適用できない場合があります。

  • 次のポイントで定義された命名規則を使用します

  • タグ <data> は noupdate=1 で更新不要なデータを設定する場合にのみ使用されます。 ファイル内に更新不要なデータがある場合は noupdate=1 は <odoo> タグに設定できます。 <data> タグには設定できません。

<record id="view_id" model="ir.ui.view">
    <field name="name">view.name</field>
    <field name="model">object_name</field>
    <field name="priority" eval="16"/>
    <field name="arch" type="xml">
        <tree>
            <field name="my_field_1"/>
            <field name="my_field_2" string="My Label" widget="statusbar" statusbar_visible="draft,sent,progress,done" />
        </tree>
    </field>
</record>

Odooはシンタックス・シュガーとして機能するカスタムタグをサポートしています:

  • menuitem: ir.ui.menu を宣言するためのショートカットとして使用します

  • template: ビューの arch セクションのみが必要な QWeb ビューを宣言するために使用します。

  • report: レポート アクション の定義に使用します。

  • act_window: レコード表記を好まない場合に使用します。

4 つの最初のタグは record 表記よりも好ましいです。

セキュリティ、ビュー、アクション¶

次のパターンを使用します。

  • メニューの場合: <model_name>_menu 、または、サブメニューの場合、<model_name>_menu_do_stuff 。

  • ビューの場合: <model_name>_view_<view_type> 。この view_type は kanban、form、tree、search、...

  • アクションの場合: メインのアクションは <model_name>_action を推奨します。その他のアクションには、サフィックス _<detail> を付け、 detail はアクションを簡潔に説明する小文字の文字列です。これは、モデルに対して複数のアクションが宣言されている場合にのみ使用されます。

  • ルールの場合: <model_name>_rule_<concerned_group> とし、この concerned_group は関係するグループの短い名前です ('model_name_group_user' の 'user' です。 パブリックユーザーなら public 、マルチカンパニーのルールなら company ) 。

XMLを継承する¶

ビューを継承する XML ID は、元のレコードと同じ ID を使用する必要があります。一目ですべての継承を見つけるのに役立ちます。 最終的な XML ID はそれらを作成するモジュールの前に追加されるため、重複はありません。

名前を見たときにオーバーライドの目的を理解しやすくするため、名前には .inherited from{details} サフィックスが含まれている必要があります。

<record id="model_view_form" model="ir.ui.view">
    <field name="name">model.view.form.inherit.module2</field>
    <field name="inherit_id" ref="module1.model_view_form"/>
    ...
</record>

新しいプライマリビューでは、最初のレコードに基づく新しいレコードなので、継承サフィックスは必要ありません。

<record id="module2.model_view_form" model="ir.ui.view">
    <field name="name">model.view.form.module2</field>
    <field name="inherit_id" ref="module1.model_view_form"/>
    <field name="mode">primary</field>
    ...
</record>

プログラミングの教養学 (Python)¶

  • 簡潔さ や言語機能やイディオムを使うよりも、常に 可読性 を好みます。

  • .clone() を使用しない

# bad
new_dict = my_dict.clone()
new_list = old_list.clone()
# good
new_dict = dict(my_dict)
new_list = list(old_list)

  • Python辞書: 生成と更新

# -- creation empty dict
my_dict = {}
my_dict2 = dict()

# -- creation with values
# bad
my_dict = {}
my_dict['foo'] = 3
my_dict['bar'] = 4
# good
my_dict = {'foo': 3, 'bar': 4}

# -- update dict
# bad
my_dict['foo'] = 3
my_dict['bar'] = 4
my_dict['baz'] = 5
# good
my_dict.update(foo=3, bar=4, baz=5)
my_dict = dict(my_dict, **my_dict2)

  • 意味のある変数/クラス/メソッド名を使用します。

  • 役に立たない変数 : 一時的な変数は、オブジェクトに名前を付けることで、コードをより明確にすることができます。 しかしそれは一時的な変数を常に作るべきではありません

# pointless
schema = kw['schema']
params = {'schema': schema}
# simpler
params = {'schema': kw['schema']}

  • 複数の戻り値は、単純な場合はOKです

# a bit complex and with a redundant temp variable
def axes(self, axis):
        axes = []
        if type(axis) == type([]):
                axes.extend(axis)
        else:
                axes.append(axis)
        return axes

 # clearer
def axes(self, axis):
        if type(axis) == type([]):
                return list(axis) # clone the axis
        else:
                return [axis] # single-element list

  • ビルドインのことを知る : Pythonビルドイン (http://docs.python.org/library/functions.html) について、少なくとも基本的な理解が必要です。

value = my_dict.get('key', None) # very very redundant
value = my_dict.get('key') # good

また、 if 'key' in my_dict と if my_dict.get('key') には大きく異なる意味があるため、正しい意味を使用していることを確認してください。

  • リスト内包を学ぶ: リスト内包、辞書内包、map 、 filter 、 sum などを利用した基本的な操作で、コードが読みやすくなります。

# not very good
cube = []
for i in res:
        cube.append((i['id'],i['name']))
# better
cube = [(i['id'], i['name']) for i in res]

  • コレクションもブール型です: pythonでは、ブール型コンテキスト(ifなど) で評価されると、多くのオブジェクトが "boolean-ish" の値を持ちます。 これらの中には、空の場合は "falsy" 、項目を含む場合は "truthy" となるコレクション (lists, dicts, sets, ...) があります。

bool([]) is False
bool([1]) is True
bool([False]) is True

ですから、if len(some_collection): の代わりに if some_collection: を書くことができます。

  • イテレータブルの反復処理

# creates a temporary list and looks bar
for key in my_dict.keys():
        "do something..."
# better
for key in my_dict:
        "do something..."
# accessing the key,value pair
for key, value in my_dict.items():
        "do something..."

  • dict.setdefault を使用する

# longer.. harder to read
values = {}
for element in iterable:
    if element not in values:
        values[element] = []
    values[element].append(other_value)

# better.. use dict.setdefault method
values = {}
for element in iterable:
    values.setdefault(element, []).append(other_value)

  • 良い開発者として、あなたのコードを文書化します (メソッドのdocstring、コードのトリッキーな部分のためのシンプルなコメント)

  • これらのガイドラインに加えて、次のリンクも興味深いでしょう: http://python.net/~gooder/projects/pycon/pycon/2007/idiomatic/handout.html (少し古いが、かなり関連性がある)

  • ジェネレータやデコレータの作成は避けてください: Odoo APIが提供するもののみを使用してください。

  • pythonのように、filtered, maped, sorted, ... メソッドを使ってコードの読み取りとパフォーマンスを簡単にします。

メソッドをバッチで機能させる¶

関数を追加する場合は、各レコードを扱うために self で繰り返すことによって複数のレコードを処理できることを確認してください。

def my_method(self)
    for record in self:
        record.do_cool_stuff()

パフォーマンス上の問題に関して、例えば、 'stat button' を開発する場合は、ループ内で search や search_count を実行しないでください。 read_group メソッドを使用して、すべての値を1つのリクエストだけで計算することをお勧めします。

def _compute_equipment_count(self):
""" Count the number of equipment per category """
    equipment_data = self.env['hr.equipment'].read_group([('category_id', 'in', self.ids)], ['category_id'], ['category_id'])
    mapped_data = dict([(m['category_id'][0], m['category_id_count']) for m in equipment_data])
    for category in self:
        category.equipment_count = mapped_data.get(category.id, 0)

コンテキストの伝搬¶

コンテキストは変更できない frozendict です。異なるコンテキストを持つメソッドを呼び出すには、with_context メソッドを使用する必要があります。

records.with_context(new_context).do_stuff() # all the context is replaced
records.with_context(**additionnal_context).do_other_stuff() # additionnal_context values override native context ones

警告

コンテキスト内でパラメータを渡すと、危険な副作用を引き起こす可能性があります。

値は自動的に伝搬されるので、予期せぬ動作が現れる可能性があります。 default_my_field キーを持つモデルの create() メソッドをコンテキストで呼び出すと、当該モデルの my_field のデフォルト値が設定されます。しかし、この作成中に、 my_field というフィールド名を持つ他のオブジェクト (sale.order.lineなど、sale.orderの作成時) が作成されると、そのデフォルト値も設定されます。

オブジェクトの動作に影響を与える重要なコンテキストを作成する必要がある場合。 良い名前を選択し、最終的にはその影響を分離するためにモジュールの名前で接頭辞を付けます。 mail モジュールのキーは良い例です: mail_create_nosubscribe, mail_notrack, mail_notify_user_signature ...

拡張可能であると考える¶

関数とメソッドは、あまりにも多くのロジックを含めるべきではありません。小さくてシンプルなメソッドをたくさん持つことは、大規模で複雑なメソッドを持つよりもお勧めします。 目安としては、メソッドに複数の責任があるとすぐに分割することです(http://en.wikipedia.org/wiki/Single_responsity_principleを参照)。

ビジネスロジックをメソッドにハードコーディングすることは、サブモジュールによる拡張が容易にできなくなるため、避けるべきです。

# do not do this
# modifying the domain or criteria implies overriding whole method
def action(self):
    ...  # long method
    partners = self.env['res.partner'].search(complex_domain)
    emails = partners.filtered(lambda r: arbitrary_criteria).mapped('email')

# better but do not do this either
# modifying the logic forces to duplicate some parts of the code
def action(self):
    ...
    partners = self._get_partners()
    emails = partners._get_emails()

# better
# minimum override
def action(self):
    ...
    partners = self.env['res.partner'].search(self._get_partner_domain())
    emails = partners.filtered(lambda r: r._filter_partners()).mapped('email')

上記のコードは、例示のために拡張しすぎていますが、可読性を考慮してトレードオフを行う必要があります。

また、関数に名前をつけてください。小さくて適切に名前を付けられた関数は、読み取り可能/メンテナンス可能なコードとより厳密なドキュメントの出発点です。

この推奨事項は、クラス、ファイル、モジュール、パッケージにも関係します(http://en.wikipedia.org/wiki/Cyclomatic_complexityも参照してください)

トランザクションをコミットしない¶

Odooフレームワークは、すべてのRPC呼び出しのトランザクションコンテキストを提供することを担当しています。 原則は、各 RPC 呼び出しの開始時に新しいデータベースカーソルが開かれることです。 応答をRPCクライアントに送信する直前に、次のように呼び出しが戻ったときにコミットします。

def execute(self, db_name, uid, obj, method, *args, **kw):
    db, pool = pooler.get_db_and_pool(db_name)
    # create transaction cursor
    cr = db.cursor()
    try:
        res = pool.execute_cr(cr, uid, obj, method, *args, **kw)
        cr.commit() # all good, we commit
    except Exception:
        cr.rollback() # error, rollback everything atomically
        raise
    finally:
        cr.close() # always close cursor opened manually
    return res

RPC呼び出しの実行中にエラーが発生した場合、トランザクションはアトミックにロールバックされ、システムの状態が保たれます。

同様に、システムはテストスイートの実行中に専用のトランザクションを提供します。 サーバーの起動オプションに応じてロールバックすることもできます

その結果、手動で cr.commit() を任意の場所で呼び出すと、非常に高い確率で、様々な方法でシステムを破壊することになります。なぜなら、部分的なコミットが発生し、その結果、部分的でクリーンでないロールバックが発生するからです。

  1. 一貫性のないビジネスデータ、通常は、データの損失です

  2. ワークフローの非同期化、ドキュメントは永久にスタックされます

  3. きれいに巻き戻すことができず、データベースを汚染し始めます そしてエラーを引き起こします(トランザクション中にエラーが発生しなくても、これは真です)

ここに非常に簡単なルールがあります。

自分のデータベースカーソルを明示的に作成 しない限り 、 cr.commit() を自分で呼び出しては いけません。そして、そのようなことが必要な状況は例外的です。

ところで、もし独自のカーソルを作成したのであれば、エラーケースや適切なロールバックを処理する必要がありますし、使い終わったらカーソルを適切に閉じる必要があります。

また、一般的な考えとは異なり、次のような状況では cr.commit() を呼び出す必要はありません。- models.Model オブジェクトの _auto_init() メソッド内: これはアドオンの初期化メソッドや、カスタムモデルを作成する際のORMのトランザクションによって処理されます - レポート内: commit() はフレームワークによって処理されるので、レポート内からでもデータベースを更新することができます - models.Transient メソッド内: これらのメソッドは、通常の models.Model のメソッドと全く同じように、トランザクション内で呼び出され、最後に対応する cr.commit()/rollback() が行われます - など。(疑問がある場合は、前述の一般的なルールを参照してください!)

今後、サーバーフレームワーク以外のすべての cr.commit() の呼び出しには、なぜそれが絶対に必要なのか、なぜそれが本当に正しいのか、なぜそれがトランザクションを壊さないのかを説明する明確なコメントが必要です。そうしないと、削除されてしまいます。

翻訳メソッドを正しく使用する¶

Odooは "アンダースコア" _( ) という名前の GetText のようなメソッドを使用して、コード内で使用されている静的文字列が実行時にコンテキストの言語を使用して翻訳される必要があることを示します。この疑似メソッドは、コード内で次のようにインポートすることでアクセスできます。

from odoo import _

このメソッドを使用する際には、いくつかの非常に重要なルールに従わなければなりません。それは、このメソッドを機能させ、無駄なジャンクで翻訳を埋め尽くすことを避けるためです。

基本的に、このメソッドはコード内で手動で書かれた静的な文字列にのみ使用し、商品名などのフィールド値を翻訳することはできません。これは、対応するフィールドの translate フラグを使用して行う必要があります。

このメソッドには、オプションで位置指定または名前指定のパラメータを指定できます。 ルールは非常にシンプルで、アンダースコア・メソッドの呼び出しは常に _('literal string') という形式で、それ以外のものは指定してはいけません。

# good: plain strings
error = _('This record is locked!')

# good: strings with formatting patterns included
error = _('Record %s cannot be modified!', record)

# ok too: multi-line literal strings
error = _("""This is a bad multiline example
             about record %s!""", record)
error = _('Record %s cannot be modified' \
          'after being validated!', record)

# bad: tries to translate after string formatting
#      (pay attention to brackets!)
# This does NOT work and messes up the translations!
error = _('Record %s cannot be modified!' % record)

# bad: formatting outside of translation
# This won't benefit from fallback mechanism in case of bad translation
error = _('Record %s cannot be modified!') % record

# bad: dynamic string, string concatenation, etc are forbidden!
# This does NOT work and messes up the translations!
error = _("'" + que_rec['question'] + "' \n")

# bad: field values are automatically translated by the framework
# This is useless and will not work the way you think:
error = _("Product %s is out of stock!") % _(product.name)
# and the following will of course not work as already explained:
error = _("Product %s is out of stock!" % product.name)

# Instead you can do the following and everything will be translated,
# including the product name if its field definition has the
# translate flag properly set:
error = _("Product %s is not available!", product.name)

また、翻訳者はアンダースコア関数に渡されたリテラル値を扱わなければならないことを念頭に置いてください。そのため、理解しやすいようにして、疑わしい文字やフォーマットを最小限に抑えるようにしてください。翻訳者は、 %s や %d 、改行などのフォーマットパターンを保存する必要があることを認識する必要がありますが、これらを分別して明白な方法で使用することが重要です。

# Bad: makes the translations hard to work with
error = "'" + question + _("' \nPlease enter an integer value ")

# Ok (pay attention to position of the brackets too!)
error = _("Answer to question %s is not valid.\n" \
          "Please enter an integer value.", question)

# Better
error = _("Answer to question %(title)s is not valid.\n" \
          "Please enter an integer value.", title=question)

一般的にOdooでは、文字列を操作する際、 .format() よりも % を優先し (文字列の中で置換する変数が1つだけの場合) 、位置指定の代わりに %(varname) を優先します (複数の変数を置換しなければならない場合) 。これにより、コミュニティの翻訳者が翻訳をしやすくなります。

記号と表記法¶

  • モデル名（ドット表記、モジュール名をプレフィックスとする） :

    • Odooモデルを定義する場合：名前の単数形を使用する（res.partnerSとsaleS.orderSの代わりにres.partnerとsale.order）。

    • Odooトランジェントを定義する場合（ウィザード）: <related_base_model>.<action> を使用してください。ここで related_base_model はトランジェントに関連するベースモデル (models/ で定義) であり、 action はトランジェントが行うことを示した短い名前です。 wizard ワードは避けてください。例： account.invoice.make 、 project.task.delegate.batch 、...

    • report モデル（SQLビューなど）を定義する際には、Transientの規約に基づき、 <related_base_model>.report.<action> を使用します。

  • Odoo Python クラス : キャメルケースを使用（オブジェクト指向スタイル）。

class AccountInvoice(models.Model):
    ...
  • 変数名：

    • モデル変数にはキャメルケースを使用

    • 共通変数にはアンダースコアの小文字表記を使用。

    • レコードのIDやIDのリストを含む場合は、変数名の最後に _id または _ids を付けます。res.partnerのレコードを含む場合は partner_id を使用しないでください。

Partner = self.env['res.partner']
partners = Partner.browse(ids)
partner_id = partners[0].id

  • One2Many および Many2Many フィールドは、常に _ids を接尾辞として持つ必要があります（例：sale_order_line_ids）。

  • Many2Oneフィールドは _id を接尾辞として持つべきである（例：partner_id, user_id, ...)

  • メソッドの規約

    • フィールドの計算：計算メソッドのパターンは _compute_<field_name> です。

    • 検索メソッド：検索メソッドのパターンは _search_<field_name> です。

    • デフォルト・メソッド：デフォルト・メソッドのパターンは _default_<field_name> です。

    • セレクション・メソッド：セレクション・メソッドのパターンは _selection_<field_name> です

    • Onchangeメソッド：onchangeメソッドのパターンは _onchange_<field_name> です。

    • Constraintメソッド：制約メソッドのパターンは _check_<constraint_name> です。

    • アクションメソッド : オブジェクトのアクションメソッドは action_ をプレフィックスとします。1つのレコードしか使わないので、メソッドの先頭に self.ensure_one() を追加します。

  • モデルの属性の順番は次のようにします。

    1. プライベート属性 (_name, _description, _inherit, ...)

    2. デフォルトメソッドと _default_get

    3. フィールド宣言

    4. フィールド宣言と同じ順序で、計算メソッド、逆計算メソッド、検索メソッド

    5. セレクション・メソッド (セレクション・フィールドの計算値を返すためのメソッド)

    6. Constrains メソッド (@api.constrains) とonchange メソッド (@api.onchange)

    7. CRUDメソッド(ORMのオーバーライド)

    8. アクション・メソッド

    9. そして最後に、その他のビジネス・メソッド。

class Event(models.Model):
    # Private attributes
    _name = 'event.event'
    _description = 'Event'

    # Default methods
    def _default_name(self):
        ...

    # Fields declaration
    name = fields.Char(string='Name', default=_default_name)
    seats_reserved = fields.Integer(string='Reserved Seats', store=True
        readonly=True, compute='_compute_seats')
    seats_available = fields.Integer(string='Available Seats', store=True
        readonly=True, compute='_compute_seats')
    price = fields.Integer(string='Price')
    event_type = fields.Selection(string="Type", selection='_selection_type')

    # compute and search fields, in the same order of fields declaration
    @api.depends('seats_max', 'registration_ids.state', 'registration_ids.nb_register')
    def _compute_seats(self):
        ...

    @api.model
    def _selection_type(self):
        return []

    # Constraints and onchanges
    @api.constrains('seats_max', 'seats_available')
    def _check_seats_limit(self):
        ...

    @api.onchange('date_begin')
    def _onchange_date_begin(self):
        ...

    # CRUD methods (and name_get, name_search, ...) overrides
    def create(self, values):
        ...

    # Action methods
    def action_validate(self):
        self.ensure_one()
        ...

    # Business methods
    def mail_user_confirm(self):
        ...

JavascriptとCSS¶

静的ファイルの構成¶

Odooのアドオンには、様々なファイルの構成方法に関する規約があります。ここでは、Webアセットがどのように構成されるべきかをより詳しく説明します。

最初に知っておくべきことは、Odooサーバーは、 static/ フォルダにあるすべてのファイルを（静的に）提供するということですが、その際、プレフィックスとしてアドオン名を付けます。例えば、あるファイルが addons/web/static/src/js/some_file.js にある場合、そのファイルは your-odoo-server.com/web/static/src/js/some_file.js という Url で静的に利用できます。

規約では、次のような構造でコードを整理しています。

  • static: すべての静的ファイル全般

    • static/lib: サブフォルダ内にjs libsを配置する場所です。例えば、 jquery ライブラリのすべてのファイルは addsons/web/static/lib/jquery にあります。

    • static/src: 一般的な静的ソースコードのフォルダです。

      • static/src/css: すべての Css ファイル

      • static/fonts

      • static/img

      • static/src/js

        • static/src/js/tours: エンドユーザーツアーファイル(テストではなくチュートリアル)

      • static/src/scss: scss ファイル

      • static/src/xml: JSでレンダリングされるすべてのqwebテンプレート

    • static/tests: ここではテスト関連のファイルをすべて配置します。

      • static/tests/tours: ここでは、ツアーのテストファイル(チュートリアルではありません) をすべて配置します。

Javascriptコーディングガイドライン¶

  • use strict; はすべての javascript ファイルに推奨されます

  • Linter (jshint, ...) を使用する

  • 圧縮されたJavascriptライブラリを追加しない

  • クラス宣言にキャメルケースを使用する

より正確な JS のガイドラインは github wiki に詳述されています。 Javascript リファレンスを参照することで、Javascript の既存の API を確認することもできます。

CSS コーディングガイドライン¶

  • すべてのクラスに o_<module_name> というプレフィックスをつけます。この module_name は、モジュールの技術的な名前 ('sale', 'im_chat', ...) 、またはモジュールが予約するメインルート ( 主にwebsiteモジュールの場合、例 : website_forum モジュールの 'o_forum') です。このルールの唯一の例外は webclient です：単に o_ プレフィックスを使用します。

  • id タグの使用を避ける

  • Bootstrapネイティブクラスを使用する

  • クラス名にはアンダースコア・小文字の表記を使う

Git¶

Git の設定¶

先祖代々の経験と口伝によれば、次のようなことをしておくとコミットをより便利にすることができます。

  • ローカルの git config で、user.email と user.name の両方を必ず定義しておきましょう。

    git config --global <var> <value>

  • Github のプロフィールには必ずフルネームを追加しましょう。あなたのチームやアバター、お気に入りの言葉などを追加してみてください ;-)

コミットメッセージの構造¶

コミットメッセージは、タグ、モジュール、短い説明、完全な説明の4つの部分で構成されています。コミットメッセージの構成は次のようにしてください。

[TAG] module: describe your change in a short sentence (ideally < 50 chars)

Long version of the change description, including the rationale for the change,
or a summary of the feature being introduced.

Please spend a lot more time describing WHY the change is being done rather
than WHAT is being changed. This is usually easy to grasp by actually reading
the diff. WHAT should be explained only if there are technical choices
or decision involved. In that case explain WHY this decision was taken.

End the message with references, such as task or bug numbers, PR numbers, and
OPW tickets, following the suggested format:
task-123 (related to task)
Fixes #123  (close related issue on Github)
Closes #123  (close related PR on Github)
opw-123 (related to ticket)

タグとモジュール名¶

タグは、コミットのプレフィックスに使用されます。次のいずれかにする必要があります。

  • [FIX] バグ修正: ほとんどが安定版で使用されていますが、開発版で最新のバグを修正している場合にも有効です。

  • [REF] リファクタリング: 機能が大幅に書き換えられたとき;

  • [ADD] 新しいモジュールを追加したため

  • [REM] リソースを削除: デッドコードを削除、ビューを削除、モジュールを削除など

  • [REV] コミットを元に戻す: コミットが問題を引き起こした場合、または復元が望まれていない場合はこのタグを使用して行われます。

  • [MOV] ファイルを移動するには: git move を使用し、移動されたファイルの内容を変更しないでください。そうしないと、Git はファイルの追跡と履歴を失う可能性があります。 あるファイルから別のファイルにコードを移動するときにも使用されます。

  • [REL] リリースコミット: 新しいメジャーまたはマイナーの安定版

  • [IMP] 改善のため：開発バージョンで行われたほとんどの変更は、他のタグとは関係のないインクリメンタルな改善です。

  • [MERGE] マージコミット: バグ修正のフォワードポートに使用されますが、複数の分割コミットを含む機能のメインコミットとしても使用されます。

  • [CLA] Odoo 個々の貢献者ライセンスに署名するため;

  • [I18N] 翻訳ファイルの変更;

タグの後には、変更されたモジュール名を記載します。機能的な名前は時間とともに変わる可能性があるので、技術的な名前を使用してください。複数のモジュールが変更されている場合は、それらをリストアップするか、クロスモジュールであることを示すために色々なものを使用してください。本当に必要な場合や簡単な場合を除いて、同じコミットで複数のモジュールにまたがるコードを修正することは避けてください。モジュールの歴史を理解するのが難しくなるかもしれません。

コミットメッセージヘッダー¶

タグとモジュール名には意味のあるコミットメッセージヘッダーが表示されます。変更の背後にある理由を説明する必要があります。 "bugfix" や "improvements" のような単語は使用しないでください。読みやすさのために、ヘッダの長さを約50文字に制限してください。

コミットメッセージヘッダは if applied, this commit will <header> と連結されると有効な文になるようにする必要があります。 例えば、 [IMP] base: prevent to archive users linked to active partners　は、 if applied, this commit will prevent users to archive... と有効な文になるので正しいです。（訳：適合された場合、このコミットは、ユーザーがアーカイブするのを防ぎます）

コミットメッセージの詳細説明¶

メッセージの説明では、変更の影響を受けるコードの部分（モジュール名、lib、transversal objectなど）と、変更内容の説明を指定します。

まず、なぜコードを変更したのかを説明します。40年後（または3日後）に誰かがあなたのコミットを見返したときに重要なのは、なぜそれを行ったのかということです。それが変更の目的です。

あなたが何をしたかは、コミット自体に書かれています。技術的な選択があった場合は、理由の後のコミットメッセージで説明するのが良いでしょう。ちなみに、OdooのR&D開発者にとっては、 "POチームに頼まれたから" という理由は有効ではありません。

複数のモジュールに同時に影響を与えるコミットは避けてください。影響を受けるモジュールが異なる場合は、異なるコミットに分けるようにしてください。これは、あるモジュールの変更を個別に戻す必要がある場合に役立ちます。

多少冗長になってもかまわないでしょう。ほとんどの人はあなたのコミットメッセージだけを見て、その数行の文章だけであなたの人生のすべてを判断するでしょう。プレッシャーは全くありません。

あなたは意味のある機能に数時間、数日、数週間かけて取り組んでいます。心を落ち着かせて、わかりやすいコミットメッセージを書きましょう。

あなたがOdoo R&Dの開発者であれば、ナゼ(WHY) はあなたが取り組んでいるタスクの目的であるべきです。完全な仕様がコミットメッセージの核となります。 もし、目的や仕様が不十分なタスクに取り組んでいる場合は、続ける前にそれらを明確にすることを検討してください。

最後に、正しいコミットメッセージの例をいくつかご紹介します。

[REF] models: use `parent_path` to implement parent_store

 This replaces the former modified preorder tree traversal (MPTT) with the
 fields `parent_left`/`parent_right`[...]

[FIX] account: remove frenglish

 [...]

 Closes #22793
 Fixes #22769

[FIX] website: remove unused alert div, fixes look of input-group-btn

 Bootstrap's CSS depends on the input-group-btn
 element being the first/last child of its parent.
 This was not the case because of the invisible
 and useless alert.

注釈

長い説明文を使って、WhatではなくWhyを説明します。Whatはdiffで見ることができます。
